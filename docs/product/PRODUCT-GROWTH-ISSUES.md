### 1.1 public 計画 726 件の帰属精査（2026-06-10）

「運営者自身のテストが大半では」という仮説を本番 DB で検証した結果、**5月以降の急増分は外部流入**だった。

| 帰属 | 件数 | 期間 | 根拠 |
|---|---|---|---|
| 登録ユーザー id=3（運営者と推定） | 22 | 2025-12〜2026-02 | 唯一の計画保有登録ユーザー |
| 匿名ユーザー id=1（開発・検証期） | 68 | 2025-12〜2026-02 | 運営者と推定 |
| user_id NULL・2〜4月 | 45 | 2026-02〜04 | 同一セッションで連続作成（最大8件/2日）など手動テストの形跡。41/45 が completed |
| **user_id NULL・5月以降（外部流入）** | **591** | 2026-05-12〜06-09 | 約580の**別々のセッション**（1セッション≒1計画）。ピークは 5/23〜31 の毎日50〜59件 |

外部判定の根拠:

- 選択地域は **us 313 / in 233 / jp 45**。上位は Williams County, ND（145）、Amritsar, Punjab（86）で日本がほぼない
- 同時期の新規登録32人は全員 gmail のインド系実名 Google アカウント（OAuth = 実在アカウント）
- Hindi（`in`）i18n・Google 広告 CV タグ（`AW-18128729350`）の存在と整合 → **インド・米国向け広告キャンペーン由来が最有力**（広告配信の実績は管理画面未観測・要確認）

### 1.2 体験フロー内の大量離脱（5月以降の591件）

| status | 件数 | 割合 |
|---|---|---|
| **pending**（作物選択まで完了・最適化未実行/未完） | **496** | **84%** |
| completed | 60 | 10% |
| failed | 22 | 4% |
| optimizing（スタックの可能性） | 13 | 2% |

pending 496 件は**全件が作物選択済み**。つまり「地域選択 → 作物選択」までは進むのに、最適化結果を見た流入ユーザーは 1 割しかいない。精査（[PRODUCTION-BENCHMARK-INVESTIGATION.md](PRODUCTION-BENCHMARK-INVESTIGATION.md) §2.5）では **373 件 = Rails 気象ジョブ途中停止**、**123 件 = Rust カットオーバー空白（最適化未デプロイ約 58h）**、**13 件 = Rust チェーン途中停止**。UX 離脱単独では説明しきれない。

**結論**: 問題は「流入がない」ことではなく、(a) **体験フローの途中（最適化前後）で 9 割が消える**、(b) **登録しても誰も個人計画を作らない（活性化 ≒ 0%）** の二段の断絶。また流入の中心がインド・米国である一方、Hindi i18n は未完成・参照データの充実度も日本中心で、**広告ターゲットと製品の準備状態がずれている**。

> 未確認: GA 上のセッション数・流入元・広告費用対効果・広告配信実績、流入が人間か bot かの最終確認（Google OAuth 通過分は実在アカウント）。

### P0 — 活性化（登録 → 最初の個人計画）

### P1 — 収益化の基盤

現状「お金を取る回路」がゼロ。ただし**活性化（P0）が直る前に課金を作っても売上は立たない**ため P1。

| # | 問題 | 根拠 | 改善案 |
|---|---|---|---|
| P1-5 | **決済・サブスク・プラン tier が存在しない** | `stripe`/`billing`/`subscription` 該当コードなし、`users` テーブルに課金カラムなし（`V1__baseline.sql`） | Stripe Checkout + Customer Portal で最小サブスク。`users` に plan tier を追加 |
| P1-6 | **リソース制限が tier 化されていない**（全員一律 Farm 4 / Crop 20 / 圃場 5） | `farm_create_limit.rs`、`crop_create_limit_policy.rs`、`fields_allocation.rs` | 既存ポリシーの上限値をユーザー tier 依存に差し替え。フリーミアム境界として最短の実装ポイント（例: 無料 = 農場1・計画2、有料 = 現行上限以上） |
| P1-7 | **特定商取引法表記がない**（有料化の法的前提） | `特商法`/`tokusho` 検索 0 件。privacy/terms はあり | 有料化と同時に `/legal/tokushoho` 追加、terms に課金条項追記 |
| P1-8 | AdSense は「ポリシー文言とフックだけ」で広告コードが未実装。現トラフィックでは収益も僅少 | `google-analytics.service.ts` の `loadAdSense()` 呼び出しのみ、定義・ユニットなし | 優先度低。公開ページ（entry-schedule / research）限定で実装するか、当面サブスクに集中して削除する（中途半端な宣言だけ残すのは負債） |
| P1-9 | API キーはマスタ CRUD 専用で商品になっていない（レート制限・課金・公開ドキュメントなし） | `masters_auth.rs`、`api_keys.rs` | 中期。最適化 API を含めた従量課金 API は B2B の選択肢として温存 |

## 3. 推奨着手順（サマリ）

```
フェーズ0（止血）     P0-0 体験フロー pending 84% の原因特定 / P0-0b 広告配信の確認・絞り込み
フェーズ1（活性化）   P0-5 計測 → P0-1/2 ログイン後導線 → P0-3/4 体験→個人計画の橋 → P0-6 LP の絞り込み
フェーズ2（継続）     P1-1 週次作業メール（P1-2 問い合わせ送信・P1-4 マジックリンクと基盤共用）
フェーズ3（収益）     P1-6 制限の tier 化 → P1-5 Stripe → P1-7 特商法
フェーズ4（集客）     P2-1〜4 SEO/シェア強化（フェーズ1 と並行可だが、活性化が直る前のスケールは非推奨）
```

**判断基準**: 北極星指標は 2 つ。(1) 体験フローの最適化完了率（現在 ≒10%）、(2) 登録ユーザーのうち private 計画を 1 件以上持つ割合（現在 1/35 ≒ 3%）。両方が改善するまで、広告投資と収益化実装は最小に抑える。
