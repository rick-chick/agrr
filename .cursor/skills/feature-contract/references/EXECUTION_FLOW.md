# Feature Contract 実行フロー

## 概要

要件から契約ドキュメントを作成し、並列開発を実現するまでのステップバイステップの手順。

## Phase 1: 要件分析と契約作成

### Step 1.1: 要件収集
```
入力: PRD、ユーザーストーリー、デザイン仕様
出力: 要件定義ドキュメント
```

1. **ビジネス要件の明確化**
   - 機能の目的と価値を理解
   - ユーザーストーリーを整理
   - 受け入れ条件を定義

2. **技術要件の整理**
   - 非機能要件（性能、セキュリティ等）
   - 既存システムとの統合要件
   - データ要件の整理

3. **ステークホルダーの確認**
   - フロントエンド開発者
   - バックエンド開発者
   - プロダクトマネージャー
   - QAエンジニア

### Step 1.2: Use Caseモデリング
```
入力: 要件定義ドキュメント
出力: Use Case図、シーケンス図
```

1. **アクターの特定**
   - Primary Actor（主要利用者）
   - Supporting Actors（補助システム）

2. **Use Caseの定義**
   - 機能の境界を明確化
   - 正常フローと異常フローを記述
   - 事前・事後条件を定義

3. **依存関係の分析**
   - 他のUse Caseとの関連
   - 共有リソースの特定
   - 優先順位付け

### Step 1.3: API契約設計
```
入力: Use Case定義
出力: API仕様書
```

1. **リソースモデリング**
   - RESTfulリソースの設計
   - URLパターンの決定
   - HTTPメソッドの選定

2. **データ構造定義**
   - Request/Responseスキーマ
   - バリデーションルール
   - エラーレスポンス仕様

3. **セキュリティ設計**
   - 認証・認可要件
   - 入力バリデーション
   - レート制限

### Step 1.4: ゴール記述への落とし込み
```
入力: API仕様、データ構造
出力: clean-architecture-goal-statement 形式のゴール記述 + 委譲プロンプト用の振る舞い要点
```

1. **テンプレート適用**
   - CONTRACT_TEMPLATE.md を整理用に参照
   - ポート/DTO・エラーケースをゴール記述に反映

2. **セルフレビュー**
   - [clean-architecture-goal-statement/SKILL.md](../../clean-architecture-goal-statement/SKILL.md) に従い ARCHITECTURE.md と突合

3. **実装開始**
   - feature-orchestrator Phase 2 へ（ゴール記述を各サブエージェントに渡す）

## Phase 2: 並列開発準備

### Step 2.1: ゴール記述の共有
```
出力: 委譲プロンプトに載せる振る舞い要点の共有
```

1. **ゴール記述の固定**
   - PR 説明または委譲プロンプトにゴール記述を含める
   - 該当テストファイル・観測可能な assert を列挙

2. **並列開発**
   - usecase-server / usecase-frontend へ同一ゴールを渡す

3. **Q&A**
   - ポート/DTO とテスト意図の不明点を解消

### Step 2.2: 開発ブランチ作成
```
出力: featureブランチの作成
```

1. **ブランチ戦略**
   - feature/[機能名]ブランチ作成
   - サーバー・フロントエンド別ブランチ

2. **CI/CD設定**
   - 自動テストの設定
   - デプロイ環境の準備

## Phase 3: UseCase層実装

### Step 3.1: Rails UseCase層
```
担当: サーバー開発者
スキル: usecase-server
```

1. **Input/Output Port定義**
   - Domain::[Feature]::Portsモジュール
   - DTOクラスの作成

2. **Gateway Interface定義**
   - Domain::[Feature]::Gatewaysモジュール
   - メソッドシグネチャの定義

3. **Interactor実装**
   - Domain::[Feature]::Interactorsクラス
   - ビジネスロジックの記述

### Step 3.2: Angular UseCase層
```
担当: フロントエンド開発者
スキル: usecase-frontend
```

1. **Input/Output Port定義**
   - TypeScript Interface
   - InjectionTokenの作成

2. **Gateway Interface定義**
   - TypeScript Interface
   - Observableベースの定義

3. **UseCase実装**
   - RxJSを使用した実装
   - エラーハンドリング

### Step 3.3: クロスレビュー
```
担当: 全員
```

1. **Interfaceの整合性確認**
   - Rails/Angular間で型定義の一致
   - DTO構造の同期

2. **テスト実装**
   - UseCase層の単体テスト
   - モックを使用した検証

## Phase 4: Adapter層実装

### Step 4.1: Rails Adapter層
```
担当: サーバー開発者
スキル: presenter-server, gateway-server, controller-server
```

1. **Presenter実装**
   - Output Portの実装
   - Viewとの連携

2. **Gateway実装**
   - Repositoryパターン
   - ActiveRecord連携

3. **Controller実装**
   - HTTPリクエスト処理
   - パラメータ変換

### Step 4.2: Angular Adapter層
```
担当: フロントエンド開発者
スキル: presenter-frontend, gateway-frontend, controller-frontend
```

1. **Presenter実装**
   - Output Portの実装
   - View State更新

2. **Gateway実装**
   - HTTP通信
   - ApiClientService使用

3. **Controller実装**
   - Component実装
   - View Interface提供

### Step 4.3: 統合テスト
```
担当: QAエンジニア
```

1. **API契約テスト**
   - リクエスト/レスポンスの検証
   - エラーケースのテスト

2. **E2Eテスト**
   - 実際のユーザー操作
   - ブラウザ自動化

## Phase 5: 統合と検証

### Step 5.1: 統合テスト実行
```
担当: 全員
```

1. **機能統合**
   - フロントエンド・バックエンドの結合
   - データフローの確認

2. **パフォーマンステスト**
   - 負荷テスト
   - 応答時間測定

3. **セキュリティテスト**
   - 認証・認可の検証
   - 脆弱性スキャン

### Step 5.2: 受け入れテスト
```
担当: プロダクトマネージャー
```

1. **受け入れ条件確認**
   - ビジネス要件の充足
   - ユーザーストーリーの実現

2. **UAT（User Acceptance Test）**
   - 実際のユーザーによるテスト
   - フィードバック収集

### Step 5.3: リリース準備
```
担当: DevOpsチーム
```

1. **デプロイメント**
   - 本番環境へのリリース
   - ロールバック計画

2. **監視設定**
   - エラートラッキング
   - パフォーマンス監視

## リスク管理

### よくあるリスクと対策

1. **契約変更のリスク**
   - **対策**: 変更管理プロセスを定義
   - **シグナル**: 要件の変更要請
   - **対応**: ステークホルダー会議、影響評価

2. **並列開発の同期ずれ**
   - **対策**: 日次スタンドアップ
   - **シグナル**: Interfaceの不整合
   - **対応**: 定期的な同期ミーティング

3. **技術的実現可能性の問題**
   - **対策**: 事前スパイク実施
   - **シグナル**: 技術的障壁の発見
   - **対応**: アーキテクチャ見直し

4. **品質劣化のリスク**
   - **対策**: 継続的インテグレーション
   - **シグナル**: テスト失敗の増加
   - **対応**: 品質ゲートの強化

## メトリクス

### 成功指標

1. **開発効率**
   - 契約作成後の並列開発割合: 目標80%以上
   - 統合時の手戻り時間: 目標5%以下

2. **品質指標**
   - 契約遵守率: 目標95%以上
   - 初回受け入れテスト通過率: 目標90%以上

3. **チーム効率**
   - クロスレビュー実施率: 目標100%
   - ドキュメント更新頻度: 変更の都度

### 測定方法

- **契約遵守率**: APIテストの合格率
- **並列開発割合**: 各Phaseでの作業重複時間
- **品質指標**: テスト結果とバグレポート
- **チーム効率**: コードレビューの実施状況

## 継続的改善

### レトロスペクティブの実施

1. **定期レビュー**
   - 毎Sprint終了時に実施
   - プロセス改善点を特定

2. **テンプレート更新**
   - 新しい知見を反映
   - ベストプラクティスの蓄積

3. **トレーニング**
   - 新メンバーのオンボーディング
   - スキル習得の支援

この実行フローに従うことで、Contract-First開発による品質の高い並列開発を実現できます。
