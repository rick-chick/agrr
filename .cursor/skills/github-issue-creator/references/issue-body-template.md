# Issue 本文テンプレート（AGRR 汎用）

`rick-chick/agrr` の GitHub issue 向け。UX/UI 監査由来は **`ux-issue-creator`** のテンプレートを使う。

## 必須セクション

```markdown
## 背景

<観測事実・再現手順・なぜ今やるか。推測語だけにしない。ファイル path / ログ / スクリーンショットを明記>

## 問題 / 目的

<何が困っているか、または何を達成したいか（1〜3 文）>

## 対応方針

1. <具体的な修正・調査の進め方>
2. <テスト方針（TDD・test-common）>
3. <ARCHITECTURE.md に触れる場合は層と禁止事項の確認>

## 完了条件

- [ ] <観測可能な完了条件（曖昧な「直す」だけにしない）>
- [ ] 関連テスト GREEN（test-common）
- [ ] <i18n なら ja / en / in 同パス更新 等、該当時のみ>

## 依存

- なし
- または #N（理由） — **未完了の前提 issue は `#番号` を必ず書く**（dispatch 依存ゲートが参照）

## 参照

- 関連 issue / PR: #N
- コード: `path/to/file`
- 規約: `ARCHITECTURE.md`（該当節）
```

## バグ向けの追記（該当時）

「問題 / 目的」の代わりに次を使ってよい。

```markdown
## 期待する動作

<…>

## 実際の動作

<…>
```

## タイトル規約

```
[P0|P1|P2] <短い要約>
[P0|P1|P2][区分] <短い要約>   # 区分は任意: bug, feat, i18n, UX, refactor, infra 等
```

| 優先度 | 目安 |
|--------|------|
| P0 | 主要フロー破綻・データ不整合・セキュリティ |
| P1 | ユーザー影響あり・早めに直したい |
| P2 | 改善・負債・見た目 |

## ラベル

| ラベル | いつ付けるか |
|--------|--------------|
| `bug` | 再現可能な不具合 |
| `enhancement` | 機能改善・UX（UX 監査一括は `ux-issue-creator`） |
| `documentation` | ドキュメントのみ |
| `agent-ready` | 本文が確定し Issue Worker に即着手させたいとき（**起票後**に付与。§6 参照） |

## 品質チェック（起票前）

- [ ] 背景に**根拠**（コード・テスト・再現）がある
- [ ] 完了条件が**観測可能**
- [ ] 1 issue = **1 スコープ**（巨大依頼は分割案を示す）
- [ ] 重複 issue を確認済み
- [ ] 依存 issue が未完了なら `## 依存` に `#N` を明記（番号必須）
