# 調査スキル・チェックリスト・禁止事項

SKILL.md で参照する進捗チェックリスト・RED 省略の根拠にしない理由・手順未適用の構造と編集前チェックの詳細。

---

## 手順未適用の構造と編集前チェック

未適用は「知らなかった」ではなく**「実行しなかった」**。同じ未適用を繰り返さないために、次の層を手続きと判断に組み込む。

| 層 | 何が起きたか | 対策 |
|----|--------------|------|
| **手続き** | エラー報告時にスキルを読まず、ステップ 5（RED による検証）を実行するトリガーがかからなかった。 | エラー報告を受けた**最初のターンで**このスキルを開き、1〜5 をこの依頼の手順として実行する。 |
| **解釈** | 「バグ修正」と「エラー調査」を分けており、後者の「RED 必須」を今回の依頼に適用しなかった。 | エラー報告＝この調査スキルの対象。RED による検証は調査完了の条件に含まれる。 |
| **優先度** | 原因が分かったらすぐ直すことを優先し、「RED 確認までやる」を後回しにした。 | 「まず RED、確認してから修正」を優先する。「原因ははっきりしている」「修正は小さい」を RED 省略の根拠にしない。 |
| **ルールの扱い** | 「禁止・ゲート」と書かれているのに、「編集前の必須チェック」として使っていなかった。 | 修正に進む前に必ず自問する: **「5（RED による検証）は完了したか？」** 未完了なら修正しない。禁止・ゲートを編集の条件として毎回チェックする。 |

**編集前の必須チェック**: ソースを編集する前に「5（RED による検証）は完了したか？」を自問し、未完了なら編集しない。調査完了の前に修正に進まない。

---

## チェックリスト（進捗用）

```
- [ ] 1. 事象を調査し、再現条件を把握した
- [ ] 2. 関連ソースを読み、データフローを追った
- [ ] 3. 候補原因を列挙した
- [ ] 4. 静的解析で候補を潰し込み、残りに優先順位を付けた
- [ ] 5. 残った原因ごとに RED が再現するテストを書き、実行して RED を確認した
- [ ] 6. cursor TODO を更新して、調査結果の検証タスクを追加した
- [ ] 7. cursor TODO を更新して、修正作業タスクを追加した
```

5 まで完了したら調査完了。以降は cursor TODO を更新して検証フェーズを開始し、検証完了後に修正フェーズを開始する。

---

## RED による検証を省略する根拠にしないこと

次の理由で「RED 用の新規テストを書かずに先に修正する」ことを根拠にしない。

| 理由 | なぜ不可か |
|------|------------|
| **事象はすでに再現できている**（422 とスタックトレースで RED はある） | 事象の再現 ≠ 「その原因が真のとき失敗するテスト」が存在すること。手順で求めるのは、**残った原因ごとに**「その原因を断ち切れば GREEN になる」テストを書くこと。エンドポイントが 422 を返すことの再現だけでは、原因とテストが 1:1 で結びついていない。 |
| **原因がほぼ特定できている**（静的解析で絞れている） | 原因が絞れていても、その原因が「真のとき失敗・偽のとき成功」するテストがコード上にないなら、修正がその原因を本当に潰したかはテストで保証されない。先にその原因用の RED テストを追加し、RED を確認してから修正する。 |
| **既存テストで確認できる**（例: `test_should_get_index` が修正後に GREEN になる） | 既存テストは「index が成功する」を検証しており、「この特定原因が解消されたこと」を検証していない。残った原因ごとに、**その原因が真なら失敗する**テストを新規に書き、そのテストの RED を確認してから調査完了とする。 |

**望ましい例**（原因が「Presenter が 'Farm not found' 以外を 422 にしている」「Gateway が Adapters::Farm を参照している」の場合）:  
先に追加する RED テストの例 — 「存在しない farm で list/fields を呼ぶと Presenter が 404 相当を返す」ことを検証するテスト、または「Gateway が ::Farm（正しい定数）を参照しているとき index が 200 を返す」ことを検証するテスト。そのテストが RED であることを確認してから調査完了とし、修正スキルで修正→GREEN 確認を行う。

---

## 手順省略を根拠にしないこと

次の理由で RED による検証を省略してはならない。

- **「早く直したい」** — テスト作成を後回しにして先に修正しない。時間短縮は RED 省略の根拠にならない。
- **「原因が明確だから」** — 原因が明確に見えても、その原因をテストで縛っていないと取り違えの可能性が残る。毎回 5（RED による検証）を実行する。
- **手順を知っていても実行時に省略する** — 手順を理解していても、実行時に「今回は」と省略すると、原因の取り違えと追加確認が発生する。

## cursor TODO ワークフロー

原因調査完了後は必ず cursor TODO を更新し、順次自走する：

1. **ステップ 6**: 調査完了後に cursor TODO を更新して「調査結果の検証」タスクを追加
2. **ステップ 7**: 検証完了後に cursor TODO を更新して「修正作業」タスクを追加

各フェーズは対応するスキルに従って実施する：
- 検証フェーズ: [investigation-verification](../../investigation-verification/SKILL.md)
- 修正フェーズ: [error-fix-red-green](../../error-fix-red-green/SKILL.md)
