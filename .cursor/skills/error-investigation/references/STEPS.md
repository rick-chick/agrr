# 各ステップの実施内容（詳細）

本 SKILL の調査ワークフロー 1〜5 の各ステップで行うことの詳細。

## 1. 事象の調査

- エラーメッセージ・スタックトレース・再現手順を正確に把握する
- 発生環境・入力・条件をメモする
- 「何が起きているか」を一文で要約する

## 2. 関連ソースの読み込み

- スタックトレースやエラー箇所から、関与するファイル・クラス・メソッドを特定する
- 呼び出し元・呼び出し先のデータフローを追う
- 必要なら code_read.mdc の観点（呼び出し階層・データ発生源）で整理する

## 3. 候補原因の洗い出し

- 考えられる原因を仮説として列挙する（優先度は後で付けてよい）
- データの発生源・変換ミス・前提の違反・境界条件などを観点にする

## 4. 候補原因の静的解析による潰し込み

- ソースの静的解析（参照・代入・型・条件分岐）で、候補を消去できるか確認する
- ログや既存テストの結果から矛盾する候補を除外する
- 残った候補に優先順位を付ける（再現しやすさ・影響範囲で判断）

## 5. RED による検証

調査完了の条件。残った原因ごとに以下を実行する。

- **テストの作成**: その原因が真なら失敗するテストを 1 本ずつ書く。テストは「その原因を断ち切れば GREEN になる」ようにする。既存のテストスキル（interactor-test-server, controller-test-server 等）に従う。
- **RED の確認**: 作成したテストを実行し、**RED（失敗）** であることを確認する。RED にならない場合は、テストが原因を捉えていないか、原因の優先順位を見直す。

**調査完了**: 原因ごとに 1 本ずつ RED が再現するテストを書き、そのテストを実行して RED であることを確認した時点で調査完了。この状態でステップ 6 に進む。

## 6. cursor TODO 更新（検証追加）

原因特定完了後に cursor TODO を更新して検証フェーズを追加する。

- **TodoWrite ツール使用**: cursor TODO を更新し、「調査結果の検証」タスクを追加
  - content: 「調査結果の検証: investigation-verification スキルに従って、REDテストの妥当性を確認する」
  - status: "pending"
- **検証フェーズの開始**: investigation-verification スキルに従って検証を実施
- **完了条件**: 検証が完了したらステップ 7 に進む

## 7. cursor TODO 更新（修正追加）

検証完了後に cursor TODO を更新して修正フェーズを追加する。

- **TodoWrite ツール使用**: cursor TODO を更新し、「修正作業」タスクを追加
  - content: 「修正作業: error-fix-red-green スキルに従って、REDテストをGREENにする」
  - status: "pending"
- **修正フェーズの開始**: error-fix-red-green スキルに従って修正を実施
- **完了条件**: REDテストがGREENになったことを確認
