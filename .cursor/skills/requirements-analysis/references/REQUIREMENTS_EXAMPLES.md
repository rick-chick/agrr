# 要件分析事例集

このドキュメントは、requirements-analysisスキルを使用した要件分析の実例をまとめています。各事例では、プロジェクト調査、要求理解、要求ブラッシュアップのプロセスを示します。

## 事例1: 病害虫管理機能の要件分析

### プロジェクト調査フェーズ

**対象機能**: 病害虫（Pest）のマスタ管理機能

**技術スタック調査結果**:
- **フロントエンド**: Angular 14.x、TypeScript、Clean Architecture
- **バックエンド**: Ruby on Rails 7.x、Clean Architecture、PostgreSQL
- **既存機能**: 作物、肥料、農薬、農作業などのマスタ管理機能が既に実装済み
- **APIパターン**: RESTful API、JSONレスポンス、JWT認証
- **データモデル**: ActiveRecordを使用、関連テーブル（crops, pesticides）との多対多関係

**既存機能の確認**:
- CRUD操作の基本パターンが確立（Create, Read, Update, Delete + List）
- 削除時にUndo機能を備えたDeletionUndoServiceが存在
- フロントエンドではUseCase層とGateway層が分離されたClean Architecture

### 要求理解フェーズ

**ユーザー要求**:
「農家が病害虫の情報を管理できるようにしたい。名前、学名、科、目、説明、発生時期、地域などの情報を登録・編集・削除できるようにしてほしい。」

**分析結果**:
- **機能要件**:
  - 病害虫の一覧表示
  - 病害虫の詳細表示
  - 病害虫の新規作成
  - 病害虫の編集
  - 病害虫の削除（Undo機能付き）

- **データ項目**:
  - name (必須): 病害虫の一般名称
  - name_scientific (任意): 学術名称
  - family (任意): 科
  - order (任意): 目
  - description (任意): 説明
  - occurrence_season (任意): 発生時期
  - region (任意): 発生地域

- **非機能要件**:
  - レスポンスタイム: ページ表示2秒以内、CRUD操作1秒以内
  - 同時接続: 想定ユーザー数50名
  - セキュリティ: APIキーベース認証

### 要求ブラッシュアップフェーズ

**既存機能との整合性確認**:
- **命名規則**: 他のマスタ機能（crop, fertilize, pesticide）と統一
- **API設計**: `/api/v1/masters/pests` パスを使用（他のマスタと同様）
- **データ構造**: 他のマスタと同様のJSONスキーマ
- **UIパターン**: 他のマスタ管理画面と同様のCRUDインターフェース

**技術的実現可能性評価**:
- **データベース**: PostgreSQLで対応可能、既存のマイグレーションパターンを使用
- **API**: RailsのRESTfulルーティングで実装可能
- **フロントエンド**: 既存のAngularコンポーネントを流用可能
- **バリデーション**: 他のマスタと同様の入力チェック

**優先順位付け**:
1. **High**: 一覧・詳細・作成・更新（基本CRUD）
2. **Medium**: 削除機能（Undo付き）
3. **Low**: 検索・フィルタ機能（将来拡張）

**リスク評価**:
- **データ整合性**: 他のテーブル（crops, pesticides）との関連付けが必要
- **パフォーマンス**: 大量データ時のページング対応
- **セキュリティ**: 適切な権限管理

### 成果物: 要件仕様書の抜粋

```markdown
# 要件仕様書: 病害虫管理機能

## 機能要件

### FR-1: 病害虫一覧表示
**概要**: 登録されている病害虫の一覧を表示する

**受け入れ条件**:
- [ ] 病害虫名、学術名、発生時期、地域が一覧表示される
- [ ] ページング機能が動作する
- [ ] 各項目の詳細画面へのリンクが存在する

### FR-2: 病害虫新規作成
**概要**: 新しい病害虫情報を登録する

**受け入れ条件**:
- [ ] 必須項目（名前）の入力チェックが機能する
- [ ] 作成成功時に一覧画面に戻る
- [ ] エラー発生時に適切なメッセージが表示される

## 非機能要件

- **性能**: 一覧表示2秒以内、CRUD操作1秒以内
- **セキュリティ**: APIキーベース認証必須
- **可用性**: 99%以上の稼働率

## 制約条件

- 既存のマスタ管理アーキテクチャに準拠
- データベースはPostgreSQLを使用
- フロントエンドはAngular/TypeScript
```

## 事例2: 農作業スケジュール機能の要件分析

### プロジェクト調査フェーズ

**対象機能**: 農作業（Agricultural Task）のスケジュール管理

**技術スタック調査結果**:
- **フロントエンド**: Angular、Ganttチャートライブラリ使用
- **バックエンド**: Rails、複雑なビジネスロジック（季節・作物による条件分岐）
- **既存機能**: 作物ステージ管理、肥料・農薬の相互作用ルール

**既存機能の確認**:
- 季節・気象条件によるスケジュール生成ロジックが存在
- Ganttチャートでの可視化機能が実装済み
- 作物別の作業パターン管理

### 要求理解フェーズ

**ユーザー要求**:
「作物の成長ステージに合わせて農作業のスケジュールを自動生成したい。季節や気象条件を考慮して、適切なタイミングで作業を提案してほしい。」

**分析結果**:
- **機能要件**:
  - 作物別の作業スケジュール自動生成
  - 季節・気象条件の考慮
  - Ganttチャートでのスケジュール表示
  - 手動調整機能

- **ビジネスルール**:
  - 作物の成長ステージによる作業タイミング
  - 気象条件による作業制限
  - 肥料・農薬の使用間隔ルール

### 要求ブラッシュアップフェーズ

**複雑さ評価**:
- **ビジネスロジック**: 季節・気象・作物ステージの複合条件
- **データ整合性**: 複数テーブルの関連付けが必要
- **パフォーマンス**: 大規模農場での計算量

**実装アプローチ**:
- スケジュール生成をバックグラウンドジョブ化
- 段階的な実装（基本機能 → 高度機能）
- テストデータの充実

## 事例3: 相互作用ルール管理機能の要件分析

### プロジェクト調査フェーズ

**対象機能**: 肥料・農薬の相互作用ルール管理

**技術スタック調査結果**:
- **バックエンド**: Rails、複雑なビジネスルールエンジン
- **データ構造**: 多対多の関係性管理
- **既存機能**: 肥料・農薬の基本マスタ管理

### 要求理解フェーズ

**ユーザー要求**:
「肥料と農薬の組み合わせによる効果や副作用を管理したい。不適切な組み合わせを警告できるようにしてほしい。」

**分析結果**:
- **機能要件**:
  - 相互作用ルールの定義（相乗効果・拮抗作用・禁止組み合わせ）
  - ルール検証機能
  - 警告表示機能

- **データモデル**:
  - 肥料-農薬の多対多関係
  - 相互作用タイプ（positive, negative, prohibited）
  - 説明文と推奨事項

## 共通パターン

### 要件分析のベストプラクティス

1. **プロジェクト調査は必須**: 既存機能・技術スタックの理解なしに要件定義できない
2. **ユーザーの言葉を機能要件に変換**: 曖昧な要求を具体的・測定可能な要件に落とし込む
3. **技術的実現可能性の早期評価**: 要求ブラッシュアップで技術的制約を明確化
4. **優先順位の明確化**: MoSCoW法などで要件の優先度を決定
5. **リスクの事前特定**: 技術的・ビジネス的なリスクを早期に洗い出し

### 成果物の品質基準

- **完全性**: すべてのステークホルダーの要求を反映
- **明確性**: 曖昧な表現を排除
- **実現可能性**: 技術的・予算的制約内で実現可能
- **テスト容易性**: 受け入れ条件がテスト可能

### 次のステップ

要件分析完了後は、以下のいずれかのアプローチで実装に進む：

1. **Contract-First開発**: feature-contractスキルでAPI契約を作成
2. **直接実装**: シンプルな機能の場合、直接usecaseスキルを使用
3. **プロトタイピング**: 不確実性の高い機能の場合、まずプロトタイプ作成

各事例では、requirements-analysisスキルの3ステップ（プロジェクト調査→要求理解→要求ブラッシュアップ）を順守し、構造化された要件定義を行っています。
